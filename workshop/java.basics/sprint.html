Object-oriented concepts are just great to help you master most coding challenges, but not all. Mais ce qui rends Java spéciale, c'est son orientation objet. A prime abord, la programmation orientée objet vous semblera juste ajouter des concepts et vous sembler un peu compliquée. At the heart of object-oriented programming, you will find essentially two concepts: classes and objects. We will cover these two concepts in details in this sprint. Au coeur de la programmation orientée-objet, vous trouverez deux concepts clés: classes et objets. Ce sprint présente ces deux concepts en détails.

Object Paradigm

We are entering the real object paradigm with this sprint. So far, we have done mostly procedural programming, via static methods and fields. This means we have been writing code almost like we would in C, with just a splash of object-oriented programming. This was a necessary first learning step, allowing to master Eclipse and some of the procedural basics of the Java programming languages. Nous entrons de plein pied dans la programmation orientée-objet avec ce sprint. Jusqu'à présent, nous avons écrit du code procédural, via des méthodes statiques et des champs statiques. Cela veut dire que nous avons écrit du code comme si nous utilisions le langage C, avec juste une pointe d'orientée-objet. Mais c'était une première étape nécessaire pour maitriser Eclipse et découvrir les bases de Java.

But object-oriented programming is really a different paradigm than your regular procedural programming with functions and types like in C. In fact, we would discourage the use of static methods and fields as much as possible in the future in order to benefit fully from the object paradigm, a paradigm that will help you master larger and more complex software. Of course, if the object paradigm is more powerful, it is also more complex to master than regular procedural programming, even though its concepts are quite simple in reality and know by all. To achieve mastery of the object paradigm will be our goal from now on. Mais la programmation orientée-objet est très différente en réalité de la programmation procédurale avec des fonctions et des types comme en C. En fait, nous découragerons l'usage des méthodes et des champs statiques dans le future pour pouvoir vraiment bénéficier pleinement de la puissance d'expression du paradigme objet. Bien sûr, parce ce paradigme est puissant, il est aussi plus difficile à maîtriser même si les concepts sont en réalité très simple et connu de tous. Achevez la maîtrise du paradigme object sera donc notre objectif à l'avenir.

The object paradigm is inspired by real life. An object is an entity, with an identity, a state, and a behavior. The identity is what sets one object apart from others. The state is what the entity knows, including references to other objects. The behavior is what it can do. So in some sense, you could imagine objects being persons, cooperating within an explicit social organization. Within a social organization, people organize themselves into groups to perform tasks, cooperating together. In coding terms, we are speaking of the object graph, with objects linked via the references they have in their state. The behavior of objects is expressed in terms of methods and objects cooperate via invoking methods on objects they know. Le paradigme objet est inspiré de la vrai vie: un objet est une entité avec une identité, un état, et un comportement. L'identité d'un objet est ce qui le rends unique, différent de tous les autres objets. L'état d'un objet est ce qu'il connaît, incluant l'identité d'autres objets. Le comportement d'un objet est ce qu'il sait faire. En quelque sorte, vous pourriez pensé en termes de personnes, coopérant au sein d'une organisation sociale explicite. Au sein de cette organisation, les individus s'organize pour effectuer les tâches à faire. En terme de programmation, nous parlons du graphe d'objets où les objets sont liés par les références qu'ils contiennent dans leur état. Le comportement d'un objet est exprimé en méthodes et les objets coopèrent en invoquant des méthodes sur les objets qu'ils connaissent.

In real life, complex social organizations call for the concept of class of people. Indeed, humans need to categorize entities to help understand the world around them. We use classes for that all the time, for example, in social organizations, we have the generic concept of a doctor, a student, a carpenter, or a baker. Each category is really a class of people that have the same knowledge and have the same skills, playing the same rôles within the social organization. All carpenters know how to work wood for exemple. We also have classes for animals like mammals versus birds. In coding terms, a class describes its instances, that is, the objects it created. So a class is a factory for objects that have a state and a behavior that corresponds to that class description of a state and behavior. Dans la vraie vie, les organisations sociales demande rapidement de penser en termes de classes d'individus. En effet, les humains ont besoin de categoriser ce qui les entourent, par exemple, nous avons le concept générique d'un médecin, d'un étudiant, d'un ébéniste, ou encore d'un boulanger. Chaque catégorie est une classe d'individus ayant les mêmes connaissances, le même comportement, jouant le même rôle au sein de l'organisation. Par exemple, tous les ébénistes savent travailler le bois. Nous utilisons les classes pour décrire le monde animal avec des classes pour les mammifères et pour les oiseaux par exemple. En termes de programmation, une classe décrit ses instances, i.e., les objets qu'elle a créé. Donc une classe est une fabrique d'objets qui ont un état et un comportement qui corresponds à la description qu'en fait cette classe.

So what is object-oriented programming? It is about designing a population of objects described by classes. In other words, it is about organizing a population of objects into an object graph, where the nodes are the objects and the edges are object references. It is also about designing classes that describe these objects in terms of fields and methods. It is finally about writing the body of these methods, with the body of a method being the actual code that will execute when that method is invoked. Qu'est-ce donc que la programmation orientée-objet? C'est la conception d'une population d'objets qui collaborent, objets qui sont décrit par un ensemble de classes. En d'autres termes, il est question d'organiser une population d'objet en un graphe d'objets, où les objets sont les noeuds et les arêtes les références entre les objets. Et du coup, il est question aussi de concevoir les classes qui décrivent ses objets, en terms de champs et de méthodes. Puis enfin, il est question d'écrire le code des méthodes.

OK, so let's get started on our journey towards mastering object-oriented programming... Et bien, c'est parti, engageons nous sur le chemin vers la maîtrise de la programmation orientée-objet...

Tasks

This task will introduce the very basics about classes and objects, using a very simple example. It will show you how to define a class, with fields and methods. It will also show you how to instantiate a class, creating objects. This task will introduce the concept of object references. Object references are the Java equivalent to C pointers. Therefore, object references are essential in organizing objects in meaningful organizations where they can cooperate to provide useful behaviors. This task will explore the concept of static fields and methods and how they differ from regular fields and methods. We will highlight also important points about references. Constructors are one of the very nice features of object-oriented programming, a feature that really helps developer write cleaner and simpler code, code that becomes easier to understand and evolve. Sounds exciting? Let's get started then! This task is an exercise in coding style, we will be contrasting object-oriented programming with regular imperative programming, C-style. Do not worry if you are not a C expert, you will be able to follow and learn from this exercise. This task introduces the concept of arrays in Java, which should feel pretty familiar to any developer. But we will use this opportunity to get you to debug a bit more, bugs that are more subtle to understand. It will be mostly about unmastered aliasing in the object graph. Code is not something you do once, execute, and then forget. Code is a living entity, changing over time. Sometimes, you start from scratch, in which case you design the code, then you debug it, and then you improve it. Sometimes, you start from existing code, which is the case in this task. This means understanding the code first and then improve it. But to improve it, you need to know what is there to improve.
This is exactly what we will do in this task. You will first understand the code given to you and you will setup an evaluation of its quality in terms of the number of objects created and the efficiency of the algorithms in place.
Based on the evaluation of the previous task, you will improve the code and evaluate the impacts of the improvements you did.
